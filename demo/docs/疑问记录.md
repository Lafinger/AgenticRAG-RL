# 疑问记录

## SFT 训练日志中的 batch / epoch / step 是什么含义？

原始日志：

```text
[INFO|trainer.py:1587] 2026-04-27 17:20:17,350 >> ***** Running training *****
[INFO|trainer.py:1588] 2026-04-27 17:20:17,350 >>   Num examples = 200
[INFO|trainer.py:1589] 2026-04-27 17:20:17,350 >>   Num Epochs = 3
[INFO|trainer.py:1590] 2026-04-27 17:20:17,350 >>   Instantaneous batch size per device = 2
[INFO|trainer.py:1593] 2026-04-27 17:20:17,350 >>   Total train batch size (w. parallel, distributed & accumulation) = 16
[INFO|trainer.py:1594] 2026-04-27 17:20:17,350 >>   Gradient Accumulation steps = 8
[INFO|trainer.py:1595] 2026-04-27 17:20:17,350 >>   Total optimization steps = 39
[INFO|trainer.py:1596] 2026-04-27 17:20:17,354 >>   Number of trainable parameters = 132,120,576
```

### 逐行解释

`Running training` 表示 HuggingFace Trainer 已经完成模型、tokenizer、数据集、LoRA 配置等准备工作，正式进入训练阶段。

`Num examples = 200` 表示本次用于 SFT 的训练样本数是 200 条。这里对应 `data\novel_eval\sft_zh_llamafactory\data.json` 中的 200 条多轮 ReAct / tool-calling 对话样本。

`Num Epochs = 3` 表示训练集会被完整遍历 3 遍。200 条样本训练 3 个 epoch，意味着模型总共会看到约 `200 × 3 = 600` 条样本次序。

`Instantaneous batch size per device = 2` 表示每张 GPU 每次前向 / 反向计算实际放入 2 条样本。当前是单卡 RTX 4090，所以单次 micro batch 就是 2 条。

`Gradient Accumulation steps = 8` 表示不会每处理 2 条样本就更新一次参数，而是连续处理 8 个 micro batch 后，把梯度累积起来再更新一次。这样可以在显存不变的情况下模拟更大的 batch。

`Total train batch size = 16` 表示一次参数更新等效看到了 16 条样本。计算公式是：

```text
total_train_batch_size =
  per_device_train_batch_size × GPU 数 × gradient_accumulation_steps

当前配置：
2 × 1 × 8 = 16
```

`Total optimization steps = 39` 表示整个训练过程中会执行 39 次参数更新。计算方式是：

```text
每个 epoch 的更新步数 = ceil(Num examples / Total train batch size)
                    = ceil(200 / 16)
                    = 13

总更新步数 = 每个 epoch 的更新步数 × epoch 数
          = 13 × 3
          = 39
```

注意，`optimization step` 不是 micro batch 次数，而是参数真正更新的次数。因为设置了 `gradient_accumulation_steps = 8`，所以约 8 次 micro batch 才对应 1 次 optimization step。

`Number of trainable parameters = 132,120,576` 表示本次真正参与训练的参数量约 1.32 亿。因为使用的是 LoRA SFT，基座模型 `Qwen3-4B-Instruct-2507` 的大部分参数被冻结，训练只更新 LoRA adapter 参数。这个数字不是 4B 全部参数，而是 LoRA 相关的可训练参数。

### 本项目这组数字的含义

这次 SFT 是一个小规模 LoRA 训练：

```text
训练样本：200 条
训练轮数：3 epoch
单卡 micro batch：2
梯度累积：8
等效 batch：16
总参数更新：39 step
可训练 LoRA 参数：132,120,576
```

因此，训练步数少是正常的，不代表训练没有启动。判断是否真的训练过，要看日志里是否从 `0/39` 开始推进、是否打印 loss，以及输出目录中 `trainer_state.json` / `train_results.json` / `adapter_model.safetensors` 是否更新。

## 修改哪些参数会影响训练速度或显存？

SFT LoRA 训练的显存和速度主要由以下几类参数决定。

### 直接影响显存的参数

`per_device_train_batch_size`：每张 GPU 单次 forward / backward 放多少条样本。它对显存影响最直接。数值越大，activation 显存越高，速度通常更快但更容易 OOM。当前是 `2`。如果 OOM，优先降到 `1`。

`cutoff_len`：每条训练样本最多保留多少 token。它对显存和速度都非常敏感，因为 attention 和 activation 都和序列长度相关。当前是 `2048`。降到 `1024` 会明显省显存、加快训练，但长样本会被截断，可能损失多轮 tool-calling 轨迹。

`gradient_checkpointing`：开启后不保存全部中间 activation，反向传播时重新计算一部分。它会明显降低显存，但训练会变慢。RTX 4090 跑 4B LoRA 时建议开启，尤其是 `cutoff_len=2048` 时。

`bf16` / `fp16`：混合精度训练会减少权重、activation 和梯度相关显存，并提升 GPU 训练速度。RTX 4090 支持 bf16，建议用 `bf16=true`。如果不启用混合精度，日志会提示 `We recommend enable mixed precision training`，显存和速度都不理想。

`lora_rank`：LoRA 的秩。rank 越大，可训练参数越多，adapter 文件越大，训练显存和计算量也更高。当前 `lora_rank=64`，本次可训练参数约 `132M`。如果想更省显存和更快，可以尝试 `32` 或 `16`，但表达能力会下降。

`lora_target`：指定给哪些线性层加 LoRA。当前是 `all`，训练参数最多，效果潜力更高，但更慢、更占显存。若只 target attention 层，显存和速度会改善，但效果可能下降。

### 主要影响速度和训练步数的参数

`num_train_epochs`：训练集重复训练几遍。显存几乎不变，但总耗时近似线性增加。当前 `3`。例如从 3 改到 1，总 step 从 39 降到 13，训练时间约降为三分之一。

`gradient_accumulation_steps`：累积多少个 micro batch 后再更新参数。它基本不增加单步显存，但会影响一次参数更新前需要跑多少次 forward / backward。当前是 `8`。在 `per_device_train_batch_size=2` 时，等效 batch 是 `16`。增大它通常会让每个 epoch 的 optimization step 变少，但总 micro batch 计算量差不多，所以不一定明显加速；它主要用于在显存有限时保持较大的等效 batch。

`save_steps`：每隔多少 optimization step 保存一次 checkpoint。保存 checkpoint 会占用磁盘并造成短暂停顿。当前 `45`，而本次总 step 只有 39，所以中途不会保存，只在结束时保存。若设置很小，会更频繁写盘，训练更慢。

`logging_steps`：每隔多少 step 打印一次日志。通常对速度影响很小。当前 `5`。

`warmup_ratio` / `warmup_steps`、`learning_rate`、`lr_scheduler_type`：主要影响训练动态和收敛质量，对显存基本没有直接影响，对速度影响也很小。

### 数据规模相关参数

训练样本数 `Num examples` 会线性影响训练总耗时。当前是 200 条。样本数翻倍，在其他参数不变时，总训练时间大致也会翻倍。

样本平均 token 长度也会影响速度和显存。即使 `cutoff_len=2048` 不变，如果数据里的多轮 tool response 很长，实际训练会更慢、更占显存。

### 本项目优先调参顺序

如果目标是避免 OOM：

```text
1. 开启 bf16=true
2. 开启 gradient_checkpointing=true
3. per_device_train_batch_size 从 2 降到 1
4. cutoff_len 从 2048 降到 1024
5. lora_rank 从 64 降到 32 或 16
```

如果目标是加快 smoke 验证：

```text
1. num_train_epochs 从 3 降到 1
2. cutoff_len 从 2048 降到 1024
3. lora_rank 从 64 降到 16 或 32
4. 使用更小基座模型，例如 Qwen3-0.6B-Instruct-2507
```

如果目标是尽量保证训练质量：

```text
1. 保持 Qwen3-4B-Instruct-2507
2. 保持 cutoff_len=2048，避免截断多轮轨迹
3. 保持 lora_rank=64
4. 使用 bf16=true 和 gradient_checkpointing=true 控制显存
5. 不盲目增加 epoch，数据只有 200 条时 epoch 太多容易过拟合
```

## `export_device=cpu` 会不会导致后面跑模型也用 CPU？

不会。

`export_device=cpu` 只影响 `llamafactory-cli export` 这一步。它表示导出 / 合并模型时，在 CPU 上加载和处理权重，然后把导出的模型文件写到 `export_dir`。这样做的优点是更省 GPU 显存，适合在显存紧张或不想占用 GPU 时导出模型。

它不决定后续推理或评测使用 CPU 还是 GPU。后面跑模型时用什么设备，取决于评测 / 推理脚本如何加载模型，例如：

```text
device_map="auto"
model.to("cuda")
LLaMA-Factory / Transformers 自动检测 cuda
命令行里是否指定 cuda / cpu
```

因此：

```text
导出阶段：export_device=cpu -> 用 CPU 导出模型
评测阶段：脚本加载模型到 cuda -> 用 GPU 推理
评测阶段：脚本加载模型到 cpu -> 用 CPU 推理
```

本项目建议：

```text
导出 base / merged model 时可以用 export_device=cpu
做模型生成评测时应使用 GPU
```

判断后续推理是否真的用了 GPU，可以在另一个 PowerShell 窗口执行：

```powershell
nvidia-smi
```

如果模型推理进程占用显存并且 GPU Util 上升，说明实际在用 GPU。

## `export_size=5` 是什么意思？

`export_size=5` 是 LLaMA-Factory 导出模型时的文件切片大小设置，通常表示每个导出的权重分片最大约 5GB。

它影响的是导出后的文件组织方式，而不是模型能力：

```text
export_size=5 -> 每个 safetensors 分片大约不超过 5GB
```

例如 4B 模型导出后可能不是一个巨大的 `model.safetensors`，而是多个分片：

```text
model-00001-of-00002.safetensors
model-00002-of-00002.safetensors
model.safetensors.index.json
```

`model.safetensors.index.json` 会记录每个参数在哪个分片里。Transformers / LLaMA-Factory 加载模型时会自动读取这些分片，用户一般不需要手动处理。

它不影响：

```text
模型参数内容
模型效果
训练结果
推理使用 CPU 还是 GPU
生成质量
```

它主要影响：

```text
单个模型文件大小
保存 / 拷贝 / 上传模型时的文件数量
某些文件系统或平台的单文件大小兼容性
```

为什么常用 `5`：

```text
1. 单个文件不会太大，便于复制和上传
2. HuggingFace 模型仓库常见分片大小就是几 GB 级别
3. Windows / 网盘 / Git LFS / 对象存储处理多个中等文件通常比一个超大文件更稳
```

如果设置更大，例如：

```text
export_size=10
```

分片数量会减少，但单个文件更大。

如果设置更小，例如：

```text
export_size=2
```

分片数量会增加，但单个文件更小。

本项目保持：

```text
export_size=5
```

即可。它是一个文件管理参数，不是训练质量参数。

## `llamafactory-cli export` 为什么报 `Please provide model_name_or_path`？

报错示例：

```text
ValueError: Please provide `model_name_or_path`.
```

这通常不是模型不存在，而是命令参数没有被 LLaMA-Factory 识别。

`llamafactory-cli train .\xxx.yaml model_name_or_path=...` 这种写法是在“已有 yaml 配置文件”的基础上做参数覆盖。也就是说：

```powershell
llamafactory-cli train .\training\sft_zh_react.yaml model_name_or_path=Qwen/Qwen3-4B-Instruct-2507
```

这里的 `model_name_or_path=...` 会被当作对 yaml 的覆盖项。

但如果执行裸 `export`，也就是没有传入 yaml 文件：

```powershell
llamafactory-cli export `
  model_name_or_path=Qwen/Qwen3-4B-Instruct-2507
```

这种 `key=value` 形式不会被当作标准命令行参数解析，因此内部会认为没有传 `model_name_or_path`。

裸 `export` 应使用带 `--` 的命令行参数：

```powershell
llamafactory-cli export `
  --model_name_or_path Qwen/Qwen3-4B-Instruct-2507 `
  --template qwen3_nothink `
  --export_dir .\models\Qwen3-4B-Instruct-2507-Base `
  --export_size 5 `
  --export_device cpu `
  --trust_remote_code true
```

另一种稳妥方式是写一个 export yaml，然后执行：

```powershell
llamafactory-cli export .\training\export_base.yaml
```

总结：

```text
有 yaml 时：可以使用 key=value 覆盖 yaml 参数
无 yaml 时：必须使用 --key value 命令行参数
```

## `--trust_remote_code true` 是什么含义？

`--trust_remote_code true` 表示允许 Transformers / LLaMA-Factory 在加载 HuggingFace 模型时，执行模型仓库里自带的 Python 代码。

普通模型只需要标准的 `config.json`、tokenizer 文件和权重文件即可加载。但有些模型会在仓库里提供自定义代码，例如：

```text
modeling_xxx.py
tokenization_xxx.py
configuration_xxx.py
```

如果模型结构、tokenizer 或特殊逻辑不是 Transformers 内置实现，就需要开启 `trust_remote_code`，否则可能加载失败。

它的作用可以理解为：

```text
trust_remote_code=false -> 只使用本地 transformers 已知的标准模型实现
trust_remote_code=true  -> 允许执行模型仓库提供的自定义实现
```

风险在于：远程模型仓库中的 Python 代码会在本机执行，因此理论上有安全风险。不要对不可信模型仓库随便开启。

对本项目的情况：

```text
模型：Qwen/Qwen3-4B-Instruct-2507
来源：Qwen 官方 HuggingFace 仓库
建议：可以开启 trust_remote_code=true
```

如果模型已经被当前 Transformers 版本原生支持，不开启也可能能跑。但在训练、导出、评测命令里显式加上：

```powershell
--trust_remote_code true
```

可以减少因为模型实现或 tokenizer 细节导致的兼容问题。

安全建议：

```text
1. 官方模型仓库可以开启
2. 来历不明的个人仓库不要随便开启
3. 企业或生产环境应固定模型 revision / commit id，避免远程代码变化
```
