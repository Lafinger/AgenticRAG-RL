# 计算资源推算

本文基于当前 `demo` 工程默认配置，估算完整复现小说域 Agentic RAG + SFT + GRPO 链路需要的算力资源。

## 结论

| 阶段 | 主要资源 | RTX 4070 Ti SUPER 16GB 是否适合 |
| --- | --- | --- |
| 数据构造 / 检索 / smoke 评测 | CPU、内存、在线模型 API token | 适合 |
| SFT LoRA | 单卡 24GB 起步，16GB 需要 QLoRA / 降 batch | 勉强，不推荐按当前配置直接跑 |
| Agentic GRPO | 推荐 24GB 以上显存先做 smoke，多卡环境更适合完整训练 | 不适合完整跑 |
| 多轮 rollout / 检索闭环 | GPU 显存、KV cache、retrieval server | 不适合本机完整跑 |

你的本机更适合承担数据流程、检索服务、Oracle trace、SFT / GRPO 数据构造和小样本 smoke。完整 `Qwen3-4B + Unsloth SFT + Unsloth GRPO` 建议放到外部 GPU 环境。

## 当前默认参数

SFT 配置来自 `training/unsloth_sft_v4.yaml`：

```yaml
model_name_or_path: unsloth/Qwen3-4B-Instruct-2507
lora_rank: 64
max_seq_length: 2048
per_device_train_batch_size: 2
gradient_accumulation_steps: 8
num_train_epochs: 3
```

当前主 SFT 数据是 tool-aware canonical ReAct v4 JSONL，包含 `full_trace`、`first_action_only`、`next_action_only` 和 `final_answer_only` 四类样本。assistant 工具轮保留从 Oracle plan 生成的短 `<think>` 和 JSON tool call，最终答案轮只保留 `<answer>`。标准训练配置使用 `sft_zh_unsloth_react_v4/train_cli.jsonl` 训练集和 `eval.jsonl` 验证集。长度统计使用项目 canonical Qwen3 ReAct renderer，因此 token 数包含 `# Tools`、`<tools>` schema、`tool` role 渲染出的 `<tool_response>` 和所有 assistant 输出。renderer 不会像 tokenizer 原生 `tools=` 模板那样给答案轮补空 `<think>`。当前 v4 CLI 训练集最长约 1885 tokens，`>2048` 为 0，主配置保持 `max_seq_length: 2048`。v4 训练时会派生 `loss_weights` 做协议边界加权监督，当前会强化 assistant 首 token、`<think>`、`</think>`、`<tool_call>`、`<answer>`、`</answer>` 和 `<|im_end|>`，并对 `</tool_call>` 降权；这不改变 token 长度，只给 collator 和 loss 计算增加很小的额外张量开销。

GRPO 配置来自 `training/unsloth_grpo.yaml`：

```text
max_prompt_length = 1024
max_completion_length = 1024
num_generations = 4
per_device_train_batch_size = 1
gradient_accumulation_steps = 8
num_train_epochs = 3
```

## 显存怎么算

模型权重显存的粗略公式：

```text
模型权重显存 = 参数量 × 每参数字节数
```

`Qwen3-4B` 使用 bf16 / fp16 时：

```text
4B × 2 bytes = 8GB
```

这只是加载权重。训练还需要梯度、优化器状态、activation、KV cache 和框架开销。

全参训练的保守显存估算口径：

```text
bf16 weights = 4B × 2 bytes = 8GB
bf16 gradients = 4B × 2 bytes = 8GB
fp32 master weights = 4B × 4 bytes = 16GB
fp32 Adam m/v = 4B × 8 bytes = 32GB
activation / buffer / cache / 框架开销 = 若干 GB
总计至少 ≈ 64GB + activation / buffer / cache / 框架开销
```

其中 `fp32 Adam m/v` 指 Adam 的一阶动量 `m` 和二阶动量 `v`，通常各 4 bytes，共 8 bytes / parameter。`fp32 master weights` 是混合精度训练中常见的高精度参数副本，用于更稳定地执行参数更新；它不要求和前向 / 反向使用的 bf16 权重精度一致。不同框架可以选择不保留 master weights、使用 bf16 optimizer、8-bit optimizer 或 ZeRO / FSDP 分片，但资源规划时按 `bf16 weights 2B + bf16 gradients 2B + fp32 master 4B + fp32 m/v 8B` 估算更保守。

LoRA SFT 不训练全部参数，optimizer 只作用于 LoRA 参数，因此显存比全参训练小很多。但 GRPO 阶段还要同时处理 actor、reference、rollout engine、vLLM KV cache 和多条采样 response，显存压力明显更大。

## GRPO 为什么需要多卡

当前 GRPO 每个训练 batch 会为每个问题生成多条 response：

```text
每步 rollout 条数 = train_batch_size × rollout.n
                 = 32 × 4
                 = 128 条 response
```

每条 response 最大长度：

```text
max_response_length = 1024 tokens
```

所以单个训练 step 最多生成：

```text
128 × 1024 = 131,072 response tokens
```

这还没有计算 prompt、tool response、多轮检索上下文、reference model、actor 训练显存和 KV cache。当前默认参数：

```text
num_generations = 4
max_completion_length = 1024
```

这说明 GPU 显存需要同时承担训练、采样和上下文缓存。建议资源：

| 配置 | 适用性 |
| --- | --- |
| 4×A100 80GB | 最稳，推荐完整复现 |
| 4×A100 40GB / 4×L40S 48GB | 可尝试，需要调小 batch、response length 或 rollout.n |
| 单卡 24GB | 只适合 SFT 或小模型 smoke，不适合完整 GRPO |
| 单卡 16GB | 只适合数据流程和小模型 smoke |

## GRPO 训练步数怎么算

设多跳 QA 数量为 `T`，验证集比例为 `val_ratio=0.1`：

```text
N_train = T × 0.9
steps_per_epoch = ceil(N_train / train_batch_size)
                = ceil(N_train / 32)
```

当前三阶段总 epoch：

```text
Stage1 = 3 epochs
Stage2 = 2 epochs
Stage3 = 1 epoch
总 epoch = 6
```

总训练 step：

```text
total_steps = steps_per_epoch × 6
```

如果 `target-count=200`：

```text
N_train = 200 × 0.9 = 180
steps_per_epoch = ceil(180 / 32) = 6
Stage1 = 6 × 3 = 18 steps
Stage2 = 6 × 2 = 12 steps
Stage3 = 6 × 1 = 6 steps
总计 = 36 steps
```

每 step rollout 128 条：

```text
总 rollout 条数 = 36 × 128 = 4,608 条
最大 response tokens = 4,608 × 1024 ≈ 4.7M tokens
```

如果 `target-count=1000`：

```text
N_train = 1000 × 0.9 = 900
steps_per_epoch = ceil(900 / 32) = 29
总 steps = 29 × 6 = 174
总 rollout 条数 = 174 × 128 = 22,272 条
最大 response tokens = 22,272 × 1024 ≈ 22.8M tokens
```

注意：这里计算的是最大 response token 数，没有把 prompt、多轮工具返回和失败重试算进去。真实训练耗时通常更高。

## SFT 训练步数怎么算

SFT 有效 batch：

```text
effective_batch = per_device_train_batch_size × gradient_accumulation_steps × GPU 数
```

单卡时：

```text
effective_batch = 2 × 8 × 1 = 16
```

SFT step 数：

```text
sft_steps = ceil(N_sft / effective_batch) × epochs
```

如果 SFT 样本数为 200：

```text
sft_steps = ceil(200 / 16) × 3
          = 13 × 3
          = 39 steps
```

SFT 数据量通常不大，主要瓶颈是 `Qwen3-4B + max_seq_length=2048 + LoRA rank=64` 的显存。

## SFT LoRA 显存怎么算

LoRA SFT 不能按全参训练估算，因为基座模型冻结，optimizer state 只作用于 LoRA adapter 参数。

粗略公式：

```text
SFT LoRA 总显存 ≈ 基座模型权重 + LoRA 参数训练态显存 + activation + 临时 buffer / 框架开销
```

### 1. 基座模型权重

`Qwen3-4B` 使用 bf16 加载：

```text
4B × 2 bytes = 8GB
```

这部分即使使用 LoRA 也必须存在，因为 forward / backward 都要经过冻结的基座模型。

### 2. LoRA 可训练参数

对一个线性层 `W ∈ R(out × in)`，LoRA 不训练原始 `W`，而是训练两个低秩矩阵：

```text
A ∈ R(r × in)
B ∈ R(out × r)
```

该层 LoRA 参数量：

```text
LoRA params = r × (in + out)
```

如果 `lora_target: all`，总 LoRA 参数量是所有目标线性层的累加：

```text
Total LoRA params = Σ r × (in_i + out_i)
```

当前配置：

```yaml
lora_rank: 64
lora_target: all
```

对 4B 级模型，LoRA 参数通常是几十 M 到一两百 M 级，远小于 4B。

### 3. LoRA 参数训练态显存

按保守口径，LoRA 可训练参数也可以按以下方式估算：

```text
LoRA bf16 weights = 2 bytes / trainable param
LoRA bf16 gradients = 2 bytes / trainable param
LoRA fp32 master weights = 4 bytes / trainable param
LoRA fp32 Adam m/v = 8 bytes / trainable param
合计 = 16 bytes / trainable param
```

如果 LoRA 参数量约为 `50M`：

```text
50M × 16 bytes = 0.8GB
```

如果 LoRA 参数量约为 `100M`：

```text
100M × 16 bytes = 1.6GB
```

这就是 LoRA 省显存的核心：不会给冻结的 4B 基座参数建立完整梯度和 Adam optimizer states。

### 4. Activation 显存

SFT LoRA 最大的不确定项通常是 activation。它主要和以下参数成正比：

```text
activation ∝ per_device_train_batch_size × sequence_length × hidden_size × layer_count
```

当前配置：

```yaml
max_seq_length: 2048
per_device_train_batch_size: 2
gradient_accumulation_steps: 8
```

注意：`gradient_accumulation_steps` 不会把单步显存乘 8。它表示连续做 8 次小 batch 后再更新参数；单步显存主要看：

```text
per_device_train_batch_size = 2
max_seq_length = 2048
```

如果开启 gradient checkpointing，activation 会明显下降，但训练会变慢。没有 checkpointing 时，activation 可能占几 GB 到十几 GB。

### 5. LoRA SFT 汇总估算

按 `Qwen3-4B + bf16 + LoRA rank 64 + batch=2 + seq=2048` 粗略估算：

| 项 | 粗略显存 |
| --- | --- |
| bf16 基座模型权重 | 8GB |
| LoRA 参数 / 梯度 / optimizer | 0.8GB - 2GB |
| activation | 6GB - 14GB，取决于 checkpointing / attention 实现 |
| 临时 buffer / CUDA / dataloader / 框架开销 | 2GB - 5GB |
| 合计 | 约 17GB - 29GB |

因此：

```text
24GB GPU：比较合理
16GB GPU：容易 OOM，除非降参数或使用 QLoRA
```

如果要在 RTX 4070 Ti SUPER 16GB 上做 SFT smoke，建议改成：

```yaml
per_device_train_batch_size: 1
cutoff_len: 1024
lora_rank: 16 或 32
gradient_checkpointing: true
```

或者使用 QLoRA：

```text
4-bit base model + LoRA
```

QLoRA 可以把基座模型权重显存从约 `8GB` 降到约 `2GB - 3GB`，16GB 会更稳。

建议：

| 配置 | 建议 |
| --- | --- |
| 1×A10 24GB / 1×L4 24GB / 1×RTX 3090 24GB | 可跑 4B LoRA SFT |
| RTX 4070 Ti SUPER 16GB | 建议改 QLoRA、`batch=1`、`cutoff_len=1024`，只做 smoke |
| 多卡 A100 | 可直接跑 SFT 并接 GRPO |

## 数据构造 API token 怎么算

当前小说 corpus 大约 366 个 chunk。

Step 3 Seed QA：

```text
调用次数 = chunk_count
        = 366 次
```

每次调用大致：

```text
输入 = prompt + chunk ≈ 1200-2000 tokens
输出 ≈ 150-300 tokens
```

按每次 1800 tokens 粗算：

```text
366 × 1800 ≈ 659K tokens
```

Step 4 多跳合并：

```text
调用次数 ≈ target-count
```

如果 `target-count=200`：

```text
200 × 800 ≈ 160K tokens
```

因此数据合成阶段在线模型 token 粗略估计：

| 规模 | 粗略 token |
| --- | --- |
| `target-count=200` | 约 80万 - 120万 tokens |
| `target-count=1000` | 约 150万 - 250万 tokens |

这部分主要消耗 API token 和等待时间，本机 CPU 即可完成。

## 推荐资源配置

如果目标是完整跑通并得到可比较实验结果：

| 资源 | 推荐 |
| --- | --- |
| 数据处理机器 | 本机即可 |
| CPU | 8 核以上 |
| 内存 | 32GB 推荐，16GB 勉强 |
| 磁盘 | 300GB 起，500GB 更稳 |
| SFT | 1×24GB GPU 起，或直接用同一台多卡机器 |
| GRPO | 4×A100 80GB 最稳 |
| 检索服务 | CPU 即可，也可和训练机同机运行 |
| 外部 API | Common 或 RightCode 用于 Step 3 / Step 4 数据合成 |

如果只做小规模验证：

```text
target-count = 50-200
SFT 使用 Qwen3-0.6B / 1.7B
GRPO 只做 print-only 或小模型 smoke
```

如果做完整 4B Agentic GRPO：

```text
Qwen3-4B + SFT + Stage1/2/3 GRPO
推荐 4×A100 80GB
```

## 本机和外部 GPU 的分工

RTX 4070 Ti SUPER 16GB 适合：

```text
文本解析
corpus / index
Seed QA 生成
多跳 QA 合成
Oracle trace
SFT / GRPO 数据构造
retrieval server
agentic smoke eval
小模型低显存 smoke
```

不适合直接跑当前默认配置的：

```text
Qwen3-4B + Unsloth GRPO + 多轮 Agentic rollout
```
