# 数据构造价格推算

本文统计 `demo` 工程中所有会调用 Doubao / 火山方舟大模型的业务，并按当前本地数据规模估算 token 量和费用。

统计快照：2026-05-01。

注意：本地 `seeds.jsonl` / `seeds.checkpoint.jsonl` 仍可能在你继续运行数据构造命令时变化。本文中的“已完成 / 待完成”是本次统计时的快照，不是永久固定值。

重要修正：`doubao-seed-2.0-pro` 是思考模型。上一版只统计了客户端最终收到的可见 `content` token，漏算了思考链 token。火山方舟深度思考文档的 usage 示例中，`output_tokens_details.reasoning_tokens` 会出现在输出统计里，并且文档明确说明“模型输出长度 = 思维链长度（如果有） + 回答长度”。因此本文现在把原来的结果标记为“可见输出 lower bound”，并新增“含思考 token 的修正估算”。

## 1. 计费模型和价格来源

当前工程默认 Doubao 模型为：

```text
doubao-seed-2-0-pro-260215
```

工程中的默认配置来自：

- `src/agentic_rag_rl/llm_client.py`
- `DEFAULT_DOUBAO_MODEL = "doubao-seed-2-0-pro-260215"`
- `DEFAULT_DOUBAO_THINKING_MODEL = "doubao-seed-2-0-pro-260215"`
- `DEFAULT_DOUBAO_JUDGE_MODEL = "doubao-seed-2-0-pro-260215"`

火山引擎官方文档中，`doubao-seed-2.0-pro` 的价格按单次请求输入长度分档。本文统计出的所有请求输入长度都远小于 32K token，因此全部按第一档计算。

价格来源：

- 火山引擎模型费用文档：<https://www.volcengine.com/docs/84458/1585097>
- 火山引擎模型版本说明中列出 `Doubao-seed-2-0-pro | 260215`：<https://www.volcengine.com/docs/84313/1350013>
- 火山方舟深度思考文档：<https://www.volcengine.com/docs/82379/1956279>

采用价格：

| 输入长度条件 | 输入价格 | 输出价格 | 本文是否命中 |
| --- | ---: | ---: | --- |
| `[0, 32]K token` | 3.2 元 / 百万 token | 16 元 / 百万 token | 是 |
| `(32, 128]K token` | 4.8 元 / 百万 token | 24 元 / 百万 token | 否 |
| `(128, 256]K token` | 9.6 元 / 百万 token | 48 元 / 百万 token | 否 |

本文使用保守口径：输入 token 全部按“未命中缓存”计算。火山引擎文档中还列出了缓存命中价格，但当前工程没有显式记录每次请求的 cache 命中情况，因此不把缓存折扣计入主估算。

对于思考模型，输出计费口径不能只看客户端最终拿到的 JSON / 文本。官方深度思考文档中的 usage 示例包含：

```json
"output_tokens_details": {
  "reasoning_tokens": 574
}
```

并且示例中 `output_tokens` 已包含 reasoning token。因此预算应按：

```text
计费输出 token = 可见回答 token + reasoning_tokens
```

当前工程的 `DoubaoLLMClient` 只返回 `choices[0].message.content`，没有把 `payload.usage` 写入 checkpoint，所以历史请求无法精确还原 reasoning token。本文用官方示例推导一个预算倍率：两个示例中 `reasoning_tokens / 可见回答 token` 约为 0.64-0.66，本文采用 **0.65 倍可见输出 token** 作为默认思考 token 估算。

费用公式：

```text
输入费用 = input_tokens / 1,000,000 * 3.2
输出费用 = output_tokens / 1,000,000 * 16
总费用 = 输入费用 + 输出费用
```

## 2. 统计口径

token 统计使用本地 `models/Qwen3-4B-Instruct-2507-Base` tokenizer 估算，流程是先把 messages 渲染成 chat template 文本，再编码计数。

这个结果适合做工程预算，但不等同于火山方舟账单的精确 token 数。真正账单应以 API 返回或平台账单中的 `prompt_tokens` / `completion_tokens` 为准。

修正后的输出 token 口径：

```text
visible_output_tokens = 本地文件中可见 JSON / 文本的 token
estimated_reasoning_tokens = visible_output_tokens * 0.65
estimated_billed_output_tokens = visible_output_tokens + estimated_reasoning_tokens
```

如果后续把 API response 的 `usage.output_tokens_details.reasoning_tokens` 记录下来，应优先使用真实 usage，而不是本文的 0.65 估算倍率。

输出 token 的处理方式：

| 业务 | 输出 token 来源 |
| --- | --- |
| KG 三元组抽取 | 已有 `triples_cache.jsonl` 的真实 JSON 输出均值，外推到未完成 chunk |
| seed QA 生成 | 已有 `seeds.jsonl` 的真实 JSON 输出均值，外推到未完成 chunk |
| 多跳 QA 合并 | 当前没有最终 `qa_pairs.jsonl`，按默认 `target_count=200` 的候选链和输出 schema 估算 |
| LLM Judge | 当前没有 judged 输出，按标准 judge JSON schema 估算 |

## 5.5 思考 token 修正

上面 5.1-5.4 的输出 token 是“可见输出 token”，即本地文件中能看到的 JSON / 文本。对于 `doubao-seed-2.0-pro`，这只是成本下界。

本文采用：

```text
reasoning_tokens ≈ visible_output_tokens * 0.65
billed_output_tokens ≈ visible_output_tokens * 1.65
```

按该口径修正主数据构造链路：

| 业务 | 输入 token | 可见输出 token | 估算 reasoning token | 估算计费输出 token | 修正后费用 |
| --- | ---: | ---: | ---: | ---: | ---: |
| KG 三元组抽取 | 5,003,674 | 1,429,870 | 929,416 | 2,359,286 | 53.760333 元 |
| seed QA 生成 | 7,964,346 | 824,278 | 535,781 | 1,360,059 | 47.246851 元 |
| 多跳 QA 合并 | 59,518 | 19,836 | 12,893 | 32,729 | 0.714122 元 |
| 主数据构造合计 | 13,027,538 | 2,273,984 | 1,478,090 | 3,752,074 | 101.721306 元 |

这张表比上一版更接近思考模型的真实账单，但仍然是估算。真实值可能因为 `reasoning.effort`、问题难度、输出长度上限、模型是否实际触发深度思考而波动。

### 5.6 按实际账单反推的校准

实际运行反馈：

```text
约 11 小时，生成 2000 多条 QA 种子，火山方舟费用约 120+ 元
```

这个账单远高于上面的 0.65 倍 reasoning 估算，说明 seed QA 任务在 `doubao-seed-2.0-pro` 默认思考模式下产生了大量不可见 reasoning token。

需要先区分“2000 多条 QA 种子”的含义：

| 口径 | 含义 | 反推结果 |
| --- | --- | --- |
| 按 2000 个 chunk 请求理解 | 每个 chunk 请求一次，最多生成 2 条 seed QA | 约 0.06 元 / 请求 |
| 按 2000 条 seed QA 记录理解 | 每个 chunk 通常生成 1-2 条 seed QA，实际请求数可能约 1000-1200 | 约 0.10-0.12 元 / 请求 |

按更保守的第一种口径反推：

```text
每个 seed chunk 请求费用 ≈ 120 / 2000 = 0.06 元
平均输入费用 ≈ 946 token / 1,000,000 * 3.2 = 0.0030 元
剩余主要是输出费用 ≈ 0.0570 元
反推计费输出 token ≈ 0.0570 / 16 * 1,000,000 ≈ 3560 token / 请求
```

而本地可见 seed 输出均值只有约 98 token / 请求。因此实际账单暗示：

```text
reasoning_tokens ≈ 3560 - 98 ≈ 3460 token / 请求
reasoning_tokens / visible_output_tokens ≈ 35 倍
```

这说明：

- 对 seed QA 这种“结构化生成 + 多规则约束”任务，思考模型可能会先进行很长的内部推理，再输出很短 JSON。
- 只按最终 JSON token 估算会严重低估成本。
- 官方示例里的 0.65 倍 reasoning 只适合说明机制，不能代表当前任务真实账单。

按实际账单校准后的 seed QA 成本：

| 范围 | 请求数 | 按 0.06 元 / 请求估算 |
| --- | ---: | ---: |
| seed QA 全量 8411 chunks | 8,411 | 504.66 元 |
| 当前本地已完成 4698 seed chunk | 4,698 | 281.88 元 |
| 从当前继续完成剩余 3713 seed chunk | 3,713 | 222.78 元 |

如果“2000 多条 QA 种子”实际指的是 seed QA 记录而不是 chunk 请求，那么每次请求费用还会更高，完整 seed QA 成本也会高于 504.66 元。

因此，本文后续总价应同时看两个口径：

- 理论 tokenizer 估算：用于理解输入 / 可见输出规模。
- 实际账单校准：用于预算，尤其是默认开启思考的 `doubao-seed-2.0-pro`。

## 7. 成本结构解释

按可见输出 lower bound，完整数据构造主链路约 78.07 元；按 0.65 倍可见输出估算 reasoning token 后，完整数据构造主链路约 101.72 元。但这两个数字都低于实际 seed QA 账单反馈。按实际 seed 账单校准，seed QA 全量单项就可能约 504.66 元；如果 KG 也出现类似 reasoning token 规模，完整主链路需要按 1000 元量级预留预算。

| 业务 | 可见输出 lower bound | 含思考 token 估算 | 修正后占比 |
| --- | ---: | ---: |
| KG 三元组抽取 | 38.889677 元 | 53.760333 元 | 52.85% |
| seed QA 生成 | 38.674355 元 | 47.246851 元 | 46.45% |
| 多跳 QA 合并 | 0.507834 元 | 0.714122 元 | 0.70% |

核心结论：

- 真正昂贵的是“按 chunk 扫全语料”的任务。
- 多跳合并本身不贵，因为只按目标 QA 数量请求。
- 输出 token 虽然少于输入 token，但输出单价是输入单价的 5 倍，因此输出格式越啰嗦，成本越明显。
- 对思考模型，隐藏 reasoning token 也会进入输出 token 统计；只看客户端拿到的 JSON 会低估成本。
- 实际账单显示 seed QA 的 reasoning token 可能达到可见输出的几十倍，默认思考模型不适合低成本批量结构化数据构造。
- 当前所有请求都在 `[0, 32]K` 输入长度档位内，不会触发更贵的长上下文档位。

## 8. 如何降低成本

优先级从高到低：

1. 减少全量 chunk 请求数。

   例如先只对高质量章节、目标小说、或抽样 chunk 生成 seed / KG。

2. 先跑 seed QA，再决定是否真的需要全量 KG。

   对多跳 QA 构造来说，seed QA 是必需链路；KG 主要增强 `graph_search`，可以分阶段生成。

3. 缩短 seed prompt 和 KG prompt。

   seed QA 的平均输入约 946 token / 请求，KG 约 595 token / 请求。prompt 每减少 100 token，全量 8411 次请求可少约 841,100 输入 token，按 3.2 元 / 百万 token 约节省 2.69 元。

4. 限制输出 JSON 的字段和长度。

   输出单价更高，KG 三元组数量、rationale、解释文本都应尽量限制。

5. 对纯结构化数据构造任务，考虑关闭或降低思考。

   火山方舟深度思考文档说明可通过 `thinking.type` 和 `reasoning.effort` 控制思考长度，`minimal` 会关闭思考并直接回答。当前工程的 `DoubaoLLMClient` 还没有传这些字段，因此使用 `doubao-seed-2.0-pro` 时应按默认会产生 reasoning token 做预算。

6. 充分使用 checkpoint。

   现有脚本已经支持断点续写。不要轻易 `--overwrite`，否则会重复请求已完成 chunk。

7. 对正式数据构造保留 `--max-concurrency`，但不要为了并发牺牲稳定性。

   并发只影响速度，不会降低 token 费用。失败重试反而会增加费用。

## 9. 建议执行顺序

如果目的是控制成本并逐步验证质量，建议：

```powershell
# 1. 先抽样生成 seed QA
uv run python .\scripts\gen_seed_qa.py `
  --corpus .\data\novel\corpus.jsonl `
  --output .\data\novel_eval\seeds.jsonl `
  --max-chunks 200 `
  --max-concurrency 2

# 2. 人工或脚本审核 seed 质量
uv run python .\scripts\clean_seed_qa.py `
  --input .\data\novel_eval\seeds.jsonl `
  --corpus .\data\novel\corpus.jsonl `
  --output .\data\novel_eval\seeds_clean.jsonl `
  --dropped-output .\data\novel_eval\seeds_dropped.jsonl

# 3. 确认质量后再扩大 seed 生成
uv run python .\scripts\gen_seed_qa.py `
  --corpus .\data\novel\corpus.jsonl `
  --output .\data\novel_eval\seeds.jsonl `
  --max-concurrency 2

# 4. 多跳合并按目标数量生成，成本较低

  --target-count 200 `
  --max-concurrency 2
```

## 10. 一句话结论

按当前 8411 个 chunk 的金庸语料规模，使用 `doubao-seed-2-0-pro-260215` 完成主数据构造链路：只看可见输出的成本下界约 **78.07 元**；按官方深度思考示例推导的 0.65 倍可见输出估算 reasoning token 后，主链路约 **101.72 元**。但你的实际账单显示，seed QA 默认思考模式可能约 **0.06 元 / chunk 请求**，全量 seed QA 单项约 **504.66 元**；如果 KG 也出现类似 reasoning token 规模，完整数据构造应按 **1000 元量级** 做预算。后续必须记录 API `usage.output_tokens_details.reasoning_tokens`，并优先考虑关闭思考或换非思考模型。
