# 训练测评

本文只记录主线 Agent loop 测评。主线模型必须在同一套 Qwen3 tools schema、项目 canonical Qwen3 ReAct renderer、assistant-only loss 和 ReAct think + JSON tool call 协议下训练与评测。

## 测评口径

主线评测入口：

```text
scripts\eval_hf_agentic.py
```

该脚本默认记录 `template=qwen3_react`，生成端不关闭 thinking，期望模型先输出与 Oracle plan 同风格的短 `<think>`，再输出 JSON `<tool_call>`。

评测集：

```text
data\novel_eval\test.jsonl
```

训练过程 eval loss 使用 `data\novel_eval\sft_zh_unsloth\eval.jsonl`，最终效果判断使用 `test.jsonl`。不要把 `test.jsonl` 放入 SFT 训练。

当前主线 ReAct SFT 要修复的关键失败形态是：

```text
<tool_call>卖饼老者 集市</tool_call>
```

这种输出不是 JSON tool payload，会被 `eval_hf_agentic.py` 计为 malformed。正确格式是 plan-driven 短 `<think>` 加 JSON tool payload：

```text
<think>要回答最终问题，先查：卖饼老者自称是哪个集市的人？</think>
<tool_call>
{"name":"keyword_search","arguments":{"query":"卖饼老者自称是哪个集市的人？"}}
</tool_call>
```

## 指标解释

| 指标 | 含义 | 目标 |
| --- | --- | --- |
| `avg_em` | 标准答案或别名 exact match | 高于 base 和旧 checkpoint |
| `avg_f1` | 答案 token F1 | 高于 base 和旧 checkpoint |
| `answer_tag_rate` | 是否稳定输出 `<answer>...</answer>` | 训练后明显上升 |
| `tool_call_rate` | 是否生成工具调用标签 | 保持可用 |
| `valid_tool_call_rate` | `<tool_call>` 内是否为合法 JSON tool payload | `>= 0.95` |
| `think_tag_rate` | 工具轮是否带短 `<think>` | `>= 0.95` |
| `malformed_think_rate` | `<think>` 标签是否残缺、重复或位置错误 | `<= 0.10` |
| `malformed_tool_fragment_rate` | 工具调用残缺或非 JSON 的比例 | `<= 0.20` |
| `multi_action_turn_rate` | 单轮生成多个 action 的比例 | `<= 0.20` |
| `max_turns_exceeded_rate` | 是否因不会停止而耗尽轮次 | 显著下降 |
| `avg_hop_recall` | 检索 chunk 是否覆盖 gold chunks | 不低于当前可用基线 |

## Base 模型测评

先测 base，确认检索和 eval 脚本本身没有坏：

```powershell
Set-Location E:\AI\AgenticRAG-RL\demo

uv run python .\scripts\eval_hf_agentic.py `
  --model unsloth/Qwen3-4B-Instruct-2507 `
  --data .\data\novel_eval\test.jsonl `
  --output .\results\sft_compare\base_agentic_50.jsonl `
  --max-samples 50
```

输出文件：

```text
results\sft_compare\base_agentic_50.jsonl
results\sft_compare\base_agentic_50_summary.json
```

## 指定 checkpoint 测评

如果要评测某个 LoRA checkpoint，建议在 PowerShell 中先解析成绝对路径，避免 PEFT 把相对 Windows 路径误判成 Hugging Face repo id：

```powershell
$adapter = (Resolve-Path .\training\outputs\unsloth_sft_qwen3_4b_lora\checkpoint-735).Path

uv run python .\scripts\eval_hf_agentic.py `
  --model unsloth/Qwen3-4B-Instruct-2507 `
  --adapter $adapter `
  --data .\data\novel_eval\test.jsonl `
  --output .\results\sft_compare\sft_agentic_ckpt735.jsonl `
  --max-samples 50
```

如果 checkpoint 存在但加载失败，优先检查目录内是否有：

```text
adapter_config.json
adapter_model.safetensors
```

如果缺少 `adapter_config.json`，该目录不是可直接加载的 PEFT adapter checkpoint，需要改用有效 checkpoint、完整 adapter 输出目录，或先导出 merged model 后再测。

## Smoke 与正式测评

每次新训练后先跑 5 条 smoke：

```powershell
uv run python .\scripts\eval_hf_agentic.py `
  --model unsloth/Qwen3-4B-Instruct-2507 `
  --adapter $adapter `
  --data .\data\novel_eval\test.jsonl `
  --output .\results\sft_compare\sft_agentic_smoke.jsonl `
  --max-samples 5
```

确认 smoke 中没有大面积格式错误后，再跑 50 条：

```powershell
uv run python .\scripts\eval_hf_agentic.py `
  --model unsloth/Qwen3-4B-Instruct-2507 `
  --adapter $adapter `
  --data .\data\novel_eval\test.jsonl `
  --output .\results\sft_compare\sft_agentic_50.jsonl `
  --max-samples 50
```

输出约定：

```text
results\sft_compare\sft_agentic_50.jsonl
results\sft_compare\sft_agentic_50_summary.json
```

## 结果审查

先看 summary：

```powershell
Get-Content .\results\sft_compare\sft_agentic_50_summary.json -Encoding UTF8
```

再抽查原始 turn：

```powershell
Get-Content .\results\sft_compare\sft_agentic_50.jsonl -Encoding UTF8 |
  Select-Object -First 3
```

重点检查：

1. `raw_turns` 中每个 assistant turn 是否只包含一个 action。
2. `<tool_call>` 内容是否是 JSON，且 `name` 属于 `keyword_search`、`dense_search`、`hybrid_search`。
3. tool response 历史是否由 `tool` role 承载，而不是被写成 assistant 或 user。
4. 证据足够后是否输出 `<answer>...</answer>`。
5. 达到最大轮次的样本，是不会停止、格式错误，还是检索证据不足。

## checkpoint 对比建议

不要只看最后一个 checkpoint。建议至少比较：

| checkpoint | 用途 |
| --- | --- |
| 早期 checkpoint | 检查工具格式是否很早学会 |
| 中期 checkpoint | 常见最佳点，可能比最后一步更稳 |
| 最后 checkpoint | 判断是否过拟合或格式退化 |

可以用统一命名输出：

```powershell
$ckpt = 735
$adapter = (Resolve-Path ".\training\outputs\unsloth_sft_qwen3_4b_lora\checkpoint-$ckpt").Path

uv run python .\scripts\eval_hf_agentic.py `
  --model unsloth/Qwen3-4B-Instruct-2507 `
  --adapter $adapter `
  --data .\data\novel_eval\test.jsonl `
  --output ".\results\sft_compare\sft_agentic_ckpt$ckpt.jsonl" `
  --max-samples 50
```

## LLM Judge

Agent loop 指标稳定后，可对输出做 LLM-as-Judge：

```powershell
uv run python .\scripts\run_llm_judge.py `
  .\results\sft_compare\sft_agentic_50.jsonl `
  --output .\results\sft_compare\sft_agentic_50_judged.json `
  --llm-provider common `
  --judge-model gpt-5.5 `
  --max-concurrency 5
```

Judge 用于辅助判断 correctness、faithfulness 和 context precision；主线格式指标仍以 `eval_hf_agentic.py` 的 summary 为准。

## 验收门槛

50 条 Agent loop 测评建议门槛：

```text
valid_tool_call_rate >= 0.95
malformed_tool_fragment_rate <= 0.20
answer_tag_rate >= 0.80
multi_action_turn_rate <= 0.20
avg_em / avg_f1 / avg_hop_recall 高于 base
```

如果 `valid_tool_call_rate` 已高但 `answer_tag_rate` 低，优先检查 finalization_only 样本和 assistant 结束 token 的 label mask。

如果 `malformed_tool_fragment_rate` 高，优先抽查 SFT 数据中是否仍存在非 JSON 的 `<tool_call>` 内容，并确认训练使用的是最新 `train_cli.jsonl`。
