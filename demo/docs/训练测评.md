# SFT 训练前后对比测评

本文记录如何比较训练前模型和 SFT LoRA 训练后模型的效果。

## 核心原则

训练前后对比必须保证以下条件一致：

- 使用同一份测试集：推荐 `data\novel_eval\test.jsonl`。
- 使用同一套 prompt / system prompt / 工具协议。
- 使用同一套解码参数，例如 `temperature`、`top_p`、`max_new_tokens`。
- 使用同一套指标，例如 EM、Token F1、答案格式成功率、工具调用格式成功率、Judge 分数。
- 训练集不要用于最终对比，避免高估 SFT 效果。

当前保留两条评测线：

- 直接答案评测：只看模型能否直接输出 `<answer>...</answer>`，用于衡量最终答案能力。
- Agent loop 评测：让模型真实发起 `<tool_call>`、读取本地检索结果并继续推理，用于衡量工具调用与检索闭环能力。

旧的 `sft_predictions.jsonl` 属于无检索环境下的单次生成。如果 prompt 要求模型调用工具，但脚本没有真正执行工具，模型很容易复读问题或工具调用文本；这不代表完整 Agent 能力。

重新训练前必须确认 SFT 使用 assistant-only loss mask：训练 loss 和 eval loss 只统计 assistant token，system/user/tool_response 全部置为 `-100`。当前 `scripts\train_sft_unsloth.py` 和 `scripts\train_sft_unsloth_trace.py` 已使用共享 mask helper；不要退回到整段 chat 都算 loss，否则模型会继续学习复读用户问题和 `<tool_response>`。

本项目中，训练前模型是：

```text
unsloth/Qwen3-4B-Instruct-2507
```

训练后模型可以有两种形态：

```text
LoRA adapter: training\outputs\unsloth_sft_qwen3_4b_lora
Merged model: models\Qwen3-4B-Instruct-2507-Unsloth-SFT-merged
```

正式测评建议使用 merged model，路径更清晰，也方便后续 GRPO 或部署。

## 先区分两类评测

### 1. 项目自带 smoke eval

`scripts\eval_agentic.py` 和 `scripts\run_cloud_eval.py` 主要验证数据、检索、证据召回和规则型 pipeline 是否闭环。它们当前不会加载 SFT 后的 Qwen 模型，因此不能直接代表“训练前后模型能力对比”。

可用于检查数据链路：

```powershell
cd E:\AI\AgenticRAG-RL\demo
uv run python .\scripts\eval_agentic.py `
  --data .\data\novel_eval\test.jsonl `
  --corpus .\data\novel\corpus.jsonl `
  --max-samples 10 `
  --output .\results\smoke_agentic_eval.json
```

### 2. 模型生成评测

训练前后模型对比应做“模型生成评测”：给 base model 和 SFT model 输入同一批问题，让模型生成答案，再对比预测和 gold answer。

最低要求：

```text
Base model 预测结果
SFT model 预测结果
同一测试集上的 EM / F1 / 格式成功率
```

更完整的要求：

```text
Base model 预测结果
SFT model 预测结果
EM / F1
<answer> 标签命中率
<tool_call> 格式命中率
Agent loop hop_recall
平均生成长度
必要时再做 LLM Judge
```

## Step 2: 选择固定测试集

推荐只用 held-out 测试集：

```text
data\novel_eval\test.jsonl
```

不要用：

```text

```

因为这些数据直接或间接参与了 SFT，不能代表泛化能力。

## Step 2.5: 固定 Base 模型

训练前 Base 可以直接使用在线模型 ID：

```text
unsloth/Qwen3-4B-Instruct-2507
```

如果已经提前保存了本地 Base 模型，也可以使用本地目录：

```text
.\models\Qwen3-4B-Instruct-2507-Base
```

两者的模型权重应相同。使用本地导出目录的好处是测评记录更稳定，也能避免后续 cache 被清理或模型 snapshot 变化导致路径不可复现。

## Step 3: 生成 Base 直接答案预测

使用训练前模型生成直接答案预测，输出到 `results\sft_compare\base_predictions.jsonl`。

命令模板：

```powershell
cd E:\AI\AgenticRAG-RL\demo
New-Item -ItemType Directory -Force .\results\sft_compare

python .\scripts\eval_hf_model.py `
  --model .\models\Qwen3-4B-Instruct-2507-Base `
  --data .\data\novel_eval\test.jsonl `
  --output .\results\sft_compare\base_predictions.jsonl `
  --template qwen3_nothink `
  --prompt-mode direct_answer `
  --max-samples 50 `
  --max-new-tokens 128 `
  --temperature 0 `
  --repetition-penalty 1.05
```

说明：`eval_hf_model.py` 会输出 JSONL 预测文件，同时自动生成同名 `_as_eval.json` 文件，后者可以直接交给 `run_llm_judge.py` 使用。

直接答案评测的 system prompt 固定要求只输出 `<answer>...</answer>`，不允许 `<tool_call>`。这里的 `--max-new-tokens 128` 只覆盖最终答案，不覆盖多轮工具调用；多轮工具调用用 Step 4.5 的 Agent loop 评测。

## Step 4: 生成 SFT 直接答案预测

使用合并后的 SFT 模型生成直接答案预测：

```powershell

  --data .\data\novel_eval\test.jsonl `
  --output .\results\sft_compare\sft_predictions.jsonl `
  --template qwen3_nothink `
  --prompt-mode direct_answer `
  --max-samples 50 `
  --max-new-tokens 128 `
  --temperature 0 `
  --repetition-penalty 1.05
```

如果不合并 LoRA，也可以评测 base + adapter：

```powershell
python .\scripts\eval_hf_model.py `
  --model unsloth/Qwen3-4B-Instruct-2507 `
  --adapter .\training\outputs\unsloth_sft_qwen3_4b_lora `
  --data .\data\novel_eval\test.jsonl `
  --output .\results\sft_compare\sft_lora_predictions.jsonl `
  --template qwen3_nothink `
  --prompt-mode direct_answer `
  --max-samples 50 `
  --max-new-tokens 128 `
  --temperature 0 `
  --repetition-penalty 1.05
```

直接答案评测只衡量最终答案输出能力，不代表工具调用能力。如果这里 F1 低，但 Agent loop 能检索到正确证据并给出答案，应以 Agent loop 的 `final EM/F1` 和 `hop_recall` 判断闭环效果。

## Step 4.5: 生成 SFT Agent loop 评测

Agent loop 评测会让模型每轮只生成一个 `<tool_call>` 或最终 `<answer>`。脚本会解析合法 JSON tool call，调用本地 `HybridRetriever.dispatch()`，把检索结果包装为 `<tool_response>` 后追加回消息历史。

```powershell
cd E:\AI\AgenticRAG-RL\demo

python .\scripts\eval_hf_agentic.py `
  --model .\models\Qwen3-4B-Instruct-2507-Unsloth-SFT-merged `
  --data .\data\novel_eval\test.jsonl `
  --corpus .\data\novel\corpus.jsonl `
  --output .\results\sft_compare\sft_agentic_eval.json `
  --template qwen3_nothink `
  --max-samples 50 `
  --max-turns 5 `
  --per-turn-max-new-tokens 256 `
  --top-k 3 `
  --temperature 0 `
  --repetition-penalty 1.05
```

输出中重点看：

```text
answer_tag_rate
valid_tool_call_rate
avg_em / avg_f1
avg_hop_recall
```

逐条样本中重点看：

```text
prediction
raw_turns
tool_calls
valid_tool_call_count
retrieved_chunk_ids
gold_chunks
evidence
em / f1 / hop_recall
```

## Step 5: 计算对比指标

对 base 和 SFT 的预测结果计算同一组指标：

```powershell
python .\scripts\compare_predictions.py `
  --base .\results\sft_compare\base_predictions.jsonl `
  --sft .\results\sft_compare\sft_predictions.jsonl `
  --output .\results\sft_compare\summary.json
```

建议统计字段：

```text
count
base_avg_em
sft_avg_em
base_avg_f1
sft_avg_f1
base_answer_tag_rate
sft_answer_tag_rate
base_tool_call_rate
sft_tool_call_rate
```

其中：

- `EM`：预测答案和标准答案完全一致。
- `Token F1`：预测答案和标准答案的 token overlap。
- `answer_tag_rate`：输出是否包含 `<answer>...</answer>`。
- `tool_call_rate`：单次生成输出是否包含 `<tool_call>{...}</tool_call>`。
- `valid_tool_call_rate`：Agent loop 中是否至少产生过一次合法工具调用。
- `hop_recall`：Agent loop 检索到的 chunk 是否覆盖 gold chunks。

SFT 的首要目标不是只提高答案 F1，而是让模型学会稳定遵守 ReAct / tool-calling / `<answer>` 协议。因此格式指标也很重要。

## Step 6: 可选 LLM-as-Judge

中文小说问答可能存在答案表述差异，EM / F1 会低估正确答案。可以在 base 和 SFT 预测上再跑真正的 LLM-as-Judge。

当前 Judge 使用所选 OpenAI-compatible 在线接口，默认 provider 为 `newapi`，默认模型为：

```text
gpt-5.5
```

运行前确认 `.env` 中已填写：

```text
NEWAPI_API_KEY=...
NEWAPI_BASE_URL=https://api.6i2.com/v1
NEWAPI_JUDGE_MODEL=gpt-5.5
```

命令模板：

```powershell
uv run python .\scripts\run_llm_judge.py `
  .\results\sft_compare\base_predictions_as_eval.json `
  --output .\results\sft_compare\base_judged.json `
  --judge-model gpt-5.5 `
  --max-concurrency 5

uv run python .\scripts\run_llm_judge.py `
  .\results\sft_compare\sft_predictions_as_eval.json `
  --output .\results\sft_compare\sft_judged.json `
  --judge-model gpt-5.5 `
  --max-concurrency 5
```

`--max-concurrency` 控制同时发起的 Judge 请求数，默认值是 `5`。如果遇到接口限流、超时或失败数上升，先降到 `1` 或 `2`。

Judge 会为每条样本输出：

```text
correctness
faithfulness
answer_format
overall
verdict
rationale
```

其中：

- `correctness`：预测答案是否与标准答案或别名语义等价。
- `faithfulness`：如果有证据文本，预测是否被证据支持；如果没有证据文本，则基于标准答案一致性做保守判断。
- `answer_format`：原始输出是否包含清晰最终答案，尤其是否遵守 `<answer>...</answer>`。
- `overall`：综合分。

注意：`run_llm_judge.py` 读取的是项目 eval JSON 结构，不是任意 JSONL 预测文件。`eval_hf_model.py` 已经会自动生成如下结构的 `_as_eval.json` 文件：

```json
{
  "summary": {},
  "results": [
    {
      "question": "...",
      "gold": "...",
      "prediction": "...",
      "evidence": [],
      "gold_chunks": [],
      "retrieved_chunk_ids": []
    }
  ]
}
```

## 推荐最终报告格式

训练前后对比结果建议整理成表格：

| 模型 | EM | F1 | `<answer>` 命中率 | `<tool_call>` 命中率 | 备注 |
| --- | --- | --- | --- | --- | --- |
| Qwen3-4B-Instruct-2507 | 待填 | 待填 | 待填 | 待填 | 训练前 |
| Qwen3-4B-Instruct-2507-Unsloth-SFT-merged | 待填 | 待填 | 待填 | 待填 | SFT 后 |

结论应优先回答：

```text
1. SFT 后是否更稳定输出 <answer>？
2. SFT 后是否更稳定输出合法 <tool_call>？
3. SFT 后 EM / F1 是否提升？
4. 如果 F1 没提升，是检索没搜到、格式错误、还是答案表述差异？
```
