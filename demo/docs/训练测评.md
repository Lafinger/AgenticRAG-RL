# 训练测评

本文只记录主线 Agent loop 测评。主线模型必须在同一套 Qwen3 tools schema、项目 canonical Qwen3 ReAct renderer、assistant-only loss 和 ReAct think + JSON tool call 协议下训练与评测。

## 测评口径

主线评测入口：

```text
scripts\eval_hf_agentic.py
```

该脚本默认记录 `template=qwen3_react`，生成端期望模型先输出与 Oracle plan 同风格的短 `<think>`，再输出 JSON `<tool_call>`。评测默认 `--assistant-start-anchor none --protocol-constraints none`，这是主基线；`--assistant-start-anchor think` 和 `--protocol-constraints strict` 只用于诊断或生产保护对照，不能替代主报告。

评测集：

```text
data\novel_eval\test.jsonl
```

训练过程 eval loss 使用 `data\novel_eval\sft_zh_unsloth_react_v4\eval.jsonl`，最终效果判断使用 `test.jsonl`。不要把 `test.jsonl` 放入 SFT 训练。

v4 要修复的关键失败形态来自 v3 主测评：无锚点主评测下 50/50 首轮只输出：

```text
</tool_call>
```

正确格式是 plan-driven 短 `<think>` 加 JSON tool payload：

```text
<think>要回答最终问题，先查：卖饼老者自称是哪个集市的人？</think>
<tool_call>
{"name":"keyword_search","arguments":{"query":"卖饼老者自称是哪个集市的人？"}}
</tool_call>
```

## 指标解释

| 指标 | 含义 | 目标 |
| --- | --- | --- |
| `avg_em` | 标准答案或别名 exact match | 高于 base 和旧 checkpoint |
| `avg_f1` | 答案 token F1 | 高于 base 和旧 checkpoint |
| `answer_tag_rate` | 是否稳定输出 `<answer>...</answer>` | `>= 0.80` |
| `valid_tool_call_rate` | 是否至少生成一次合法 ReAct JSON tool call | `>= 0.95` |
| `think_tag_rate` | 工具轮是否带短 `<think>` | `>= 0.95` |
| `malformed_think_rate` | `<think>` 标签是否残缺、重复或位置错误 | `<= 0.10` |
| `malformed_tool_fragment_rate` | 工具调用残缺或非 JSON 的比例 | `<= 0.20` |
| `starts_with_closing_tool_rate` | 是否首轮或某轮以 `</tool_call>` 开始 | `<= 0.05` |
| `multi_action_turn_rate` | 单轮生成多个 action 的比例 | `<= 0.20` |
| `max_turns_exceeded_rate` | 是否因不会停止而耗尽轮次 | 显著下降 |
| `avg_hop_recall` | 检索 chunk 是否覆盖 gold chunks | 高于当前失败基线 |

## Base 模型测评

先测 base，确认检索和 eval 脚本本身没有坏：

```powershell
uv run python `
  ./scripts/eval_hf_agentic.py `
  --model unsloth/Qwen3-4B-Instruct-2507 `
  --data ./data/novel_eval/test.jsonl `
  --output ./results/sft_compare/base_agentic_50.jsonl `
  --max-samples 50
```

输出文件：

```text
results\sft_compare\base_agentic_50.jsonl
results\sft_compare\base_agentic_50_summary.json
```

## 指定 checkpoint 测评

如果要评测某个 LoRA checkpoint，直接把 checkpoint 目录传给 `--adapter`：

```powershell
uv run python `
  ./scripts/eval_hf_agentic.py `
  --model unsloth/Qwen3-4B-Instruct-2507 `
  --adapter ./training/outputs/unsloth_sft_qwen3_4b_lora_react_v4/checkpoint-735 `
  --data ./data/novel_eval/test.jsonl `
  --output ./results/sft_compare/sft_agentic_react_v4_ckpt735.jsonl `
  --max-samples 50
```

`eval_hf_agentic.py` 会把存在的本地路径解析成绝对路径，避免 PEFT 把相对 Windows 路径误判成 Hugging Face repo id。如果 checkpoint 存在但加载失败，优先检查目录内是否有：

```text
adapter_config.json
adapter_model.safetensors
```

如果缺少 `adapter_config.json`，该目录不是可直接加载的 PEFT adapter checkpoint，需要改用有效 checkpoint、完整 adapter 输出目录，或先导出 merged model 后再测。

## Anchor 对照测评

主报告必须使用默认 anchor：

```text
--assistant-start-anchor none
```

如需判断是否只是 assistant 起始 token 不稳，可额外跑诊断对照：

```powershell
uv run python `
  ./scripts/eval_hf_agentic.py `
  --model unsloth/Qwen3-4B-Instruct-2507 `
  --adapter ./training/outputs/unsloth_sft_qwen3_4b_lora_react_v4/checkpoint-735 `
  --data ./data/novel_eval/test.jsonl `
  --output ./results/sft_compare/sft_agentic_react_v4_ckpt735_anchor_think.jsonl `
  --max-samples 50 `
  --assistant-start-anchor think
```

summary 会记录 `assistant_start_anchor`。anchor 对照如果明显更好，说明模型主要问题仍是首轮起始状态；仍需要回到主线数据或训练策略修复。

## Strict 约束对照测评

v4 评测脚本可选 `--protocol-constraints strict`，用于生产保护或定位“只差首 token 边界”的情况。它会在 assistant 新 turn 起始位置禁止 `</tool_call>`、`<tool_call>`、`</think>` 和 `<|im_end|>` 等非法首 token，但这不是主线验收口径。

```powershell
uv run python `
  ./scripts/eval_hf_agentic.py `
  --model unsloth/Qwen3-4B-Instruct-2507 `
  --adapter ./training/outputs/unsloth_sft_qwen3_4b_lora_react_v4/checkpoint-735 `
  --data ./data/novel_eval/test.jsonl `
  --output ./results/sft_compare/sft_agentic_react_v4_ckpt735_strict.jsonl `
  --max-samples 50 `
  --protocol-constraints strict
```

summary 会记录 `protocol_constraints`。如果 strict 对照明显好于主评测，说明模型协议边界仍没靠训练本身学稳，不能用 strict 结果冒充主结果。

## 首 Token Probe

如果 5 条或 50 条测评仍出现首轮 `</tool_call>`，不要只看生成文本；还要检查模型在训练样本 prompt 上的首 token 概率：

```powershell
uv run python `
  ./scripts/diagnose_sft_protocol.py `
  --config ./training/unsloth_sft_v4.yaml `
  --max-samples 50 `
  --probe-model unsloth/Qwen3-4B-Instruct-2507 `
  --probe-adapter ./training/outputs/unsloth_sft_qwen3_4b_lora_react_v4/checkpoint-735 `
  --probe-max-prompts 20
```

`first_token_probe.top1_closing_tool_rate` 如果仍高，说明问题在训练目标或 checkpoint 本身；如果 probe 正常但 Agent loop 异常，再优先排查评测 prompt、history 写入和 stop criteria。

## Smoke 与正式测评

每次新训练后先跑 5 条 smoke：

```powershell
uv run python `
  ./scripts/eval_hf_agentic.py `
  --model unsloth/Qwen3-4B-Instruct-2507 `
  --adapter ./training/outputs/unsloth_sft_qwen3_4b_lora_react_v4/checkpoint-735 `
  --data ./data/novel_eval/test.jsonl `
  --output ./results/sft_compare/sft_agentic_react_v4_smoke.jsonl `
  --max-samples 5
```

如果 smoke 中 `starts_with_closing_tool_rate <= 0.05`、`think_tag_rate >= 0.90` 且 `valid_tool_call_rate >= 0.80`，再跑 50 条：

```powershell

  ./scripts/eval_hf_agentic.py `
  --model unsloth/Qwen3-4B-Instruct-2507 `
  --adapter ./training/outputs/unsloth_sft_qwen3_4b_lora_react_v4/checkpoint-735 `
```

## 结果审查

先看 summary：

```powershell
uv run python `
  -m json.tool `
  ./results/sft_compare/sft_agentic_react_v4_50_summary.json
```

再抽查原始 turn：

```powershell
$code = @'
from pathlib import Path
lines = Path('./results/sft_compare/sft_agentic_react_v4_50.jsonl').read_text(encoding='utf-8').splitlines()
print('\n'.join(lines[:3]))
'@
uv run python `
  -c $code
```

重点检查：

1. `raw_turns` 中每个 assistant turn 是否只包含一个 action。
2. `<tool_call>` 内容是否是 JSON，且 `name` 属于 `keyword_search`、`dense_search`、`hybrid_search`。
3. 工具轮是否先有短 `<think>`，再有 `<tool_call>`。
4. tool response 历史是否由 `tool` role 承载，而不是被写成 assistant。
5. 证据足够后是否输出 `<answer>...</answer>`。
6. 达到最大轮次的样本，是不会停止、格式错误，还是检索证据不足。

## checkpoint 对比建议

不要只看最后一个 checkpoint。建议至少比较：

| checkpoint | 用途 |
| --- | --- |
| 早期 checkpoint | 检查工具格式是否很早学会 |
| 中期 checkpoint | 常见最佳点，可能比最后一步更稳 |
| 最后 checkpoint | 判断是否过拟合或格式退化 |

可以用统一命名输出，例如 checkpoint 735：

```powershell
uv run python `
  ./scripts/eval_hf_agentic.py `
  --model unsloth/Qwen3-4B-Instruct-2507 `
  --adapter ./training/outputs/unsloth_sft_qwen3_4b_lora_react_v4/checkpoint-735 `
  --data ./data/novel_eval/test.jsonl `
  --output ./results/sft_compare/sft_agentic_react_v4_ckpt735.jsonl `
  --max-samples 50
```

## LLM Judge

Agent loop 指标稳定后，可对输出做 LLM-as-Judge：

```powershell
uv run python `
  ./scripts/run_llm_judge.py `
  ./results/sft_compare/sft_agentic_react_v4_50.jsonl `
  --output ./results/sft_compare/sft_agentic_react_v4_50_judged.json `
  --llm-provider common `
  --judge-model gpt-5.5 `
  --max-concurrency 5
```

Judge 用于辅助判断 correctness、faithfulness 和 context precision；主线格式指标仍以 `eval_hf_agentic.py` 的 summary 为准。

## 验收门槛

50 条 Agent loop 测评建议门槛：

```text
think_tag_rate >= 0.95
valid_tool_call_rate >= 0.95
answer_tag_rate >= 0.80
malformed_tool_fragment_rate <= 0.20
starts_with_closing_tool_rate <= 0.05
multi_action_turn_rate <= 0.20
avg_em / avg_f1 / avg_hop_recall 高于当前失败基线
```

如果 `starts_with_closing_tool_rate` 高，优先确认使用的是 v4 数据目录、v4 LoRA 输出目录和 weighted protocol loss。旧 v3 checkpoint 已确认不是有效主基线。

如果 `valid_tool_call_rate` 已高但 `answer_tag_rate` 低，优先检查 `final_answer_only` 样本和 assistant 结束 token 的 label mask。

如果 `malformed_tool_fragment_rate` 高，优先抽查 SFT 数据中是否仍存在非 JSON 的 `<tool_call>` 内容，并确认训练使用的是最新 `train_cli.jsonl`。
