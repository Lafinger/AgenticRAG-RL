# SFT 训练前后对比测评

本文记录如何比较训练前模型和 SFT LoRA 训练后模型的效果。

## 核心原则

训练前后对比必须保证以下条件一致：

- 使用同一份测试集：推荐 `data\novel_eval\test.jsonl`。
- 使用同一套 prompt / system prompt / 工具协议。
- 使用同一套解码参数，例如 `temperature`、`top_p`、`max_new_tokens`。
- 使用同一套指标，例如 EM、Token F1、答案格式成功率、工具调用格式成功率、Judge 分数。
- 训练集不要用于最终对比，避免高估 SFT 效果。

当前保留两条评测线：

- 直接答案评测：只看模型能否直接输出 `<answer>...</answer>`，用于衡量最终答案能力。
- Agent loop 评测：让模型真实发起 `<tool_call>`、读取本地检索结果并继续推理，用于衡量工具调用与检索闭环能力。

双轨修复后不要再把两条评测线混成一个结论：

- `models\Qwen3-4B-Instruct-2507-Unsloth-SFT-merged` 是 Agentic SFT 模型，优先用于 Agent loop 工具调用闭环。
- `models\Qwen3-4B-Instruct-2507-Unsloth-SFT-direct-merged` 是 direct-answer 校准模型，优先用于无检索直接答案评测。

旧的 `results\sft_compare\sft_predictions.jsonl` 属于用 Agentic SFT 模型做 direct-answer 单次生成的诊断结果。如果该文件出现 `<answer>` 缺失、残缺 `<tool_call>` 或 F1 接近 0，不能直接否定完整 Agent 能力；应改用 direct-answer 校准模型生成新的 `sft_direct_predictions.jsonl`，同时单独跑 Agent loop 评测。

重新训练前必须确认 SFT 使用 assistant-only loss mask：训练 loss 和 eval loss 只统计 assistant token，system/user/tool_response 全部置为 `-100`。当前 `scripts\train_sft_unsloth.py` 和 `scripts\train_sft_unsloth_trace.py` 已使用共享 mask helper；不要退回到整段 chat 都算 loss，否则模型会继续学习复读用户问题和 `<tool_response>`。

本项目中，训练前模型是：

```text
unsloth/Qwen3-4B-Instruct-2507
```

训练后模型可以有两种形态：

```text
LoRA adapter: training\outputs\unsloth_sft_qwen3_4b_lora
Merged model: models\Qwen3-4B-Instruct-2507-Unsloth-SFT-merged
```

direct-answer 校准模型使用独立产物，不覆盖 Agentic SFT 模型：

```text
Direct-answer LoRA adapter: training\outputs\unsloth_sft_qwen3_4b_direct_lora
Direct-answer merged model: models\Qwen3-4B-Instruct-2507-Unsloth-SFT-direct-merged
Direct-answer SFT data: data\novel_eval\sft_direct_answer\train.jsonl / eval.jsonl
```

Agent loop 闭环校准模型同样使用独立产物，不覆盖原 Agentic SFT 模型：

```text
Agentic-stop LoRA adapter: training\outputs\unsloth_sft_qwen3_4b_agentic_stop_lora
Agentic-stop merged model: models\Qwen3-4B-Instruct-2507-Unsloth-SFT-agentic-stop-merged
Agentic-stop SFT data: data\novel_eval\sft_agentic_stop\train.jsonl / eval.jsonl
```

正式测评建议使用 merged model，路径更清晰，也方便后续 GRPO 或部署。Agent loop 使用 Agentic merged model；direct-answer 使用 direct merged model。

## 先区分两类评测

### 1. 项目自带 smoke eval

`scripts\eval_agentic.py` 和 `scripts\run_cloud_eval.py` 主要验证数据、检索、证据召回和规则型 pipeline 是否闭环。它们当前不会加载 SFT 后的 Qwen 模型，因此不能直接代表“训练前后模型能力对比”。

可用于检查数据链路：

```powershell
cd E:\AI\AgenticRAG-RL\demo
uv run python .\scripts\eval_agentic.py `
  --data .\data\novel_eval\test.jsonl `
  --corpus .\data\novel\corpus.jsonl `
  --max-samples 10 `
  --output .\results\smoke_agentic_eval.json
```

### 2. 模型生成评测

训练前后模型对比应做“模型生成评测”：给 base model 和 SFT model 输入同一批问题，让模型生成答案，再对比预测和 gold answer。

最低要求：

```text
Base model 预测结果
SFT model 预测结果
同一测试集上的 EM / F1 / 格式成功率
```

更完整的要求：

```text
Base model 预测结果
SFT model 预测结果
EM / F1
<answer> 标签命中率
direct-answer tool_fragment_rate
Agent loop valid_tool_call_rate / hop_recall
平均生成长度
必要时再做 LLM Judge
```

## Step 2.9: 构造并训练 Agentic-stop 闭环校准模型

如果 Agent loop 评测显示模型会调用工具但不会停止输出 `<answer>`，先构造 agentic-stop 校准数据。该链路保留工具调用能力，但补强“检索后停止并回答”的监督。

构造 agentic-stop 数据：

```powershell
cd E:\AI\AgenticRAG-RL\demo

uv run python .\scripts\build_agentic_stop_sft.py
```

默认输入和输出：

| 类型 | 路径 | 说明 |
| --- | --- | --- |
| train 输入 | `data\novel_eval\sft_zh_unsloth\train_studio.jsonl` | 已切分的 Agentic SFT train |
| eval 输入 | `data\novel_eval\sft_zh_unsloth\eval.jsonl` | 已切分的 Agentic SFT eval |
| train 输出 | `data\novel_eval\sft_agentic_stop\train.jsonl` | 规范完整轨迹 + finalization-only 样本 |
| eval 输出 | `data\novel_eval\sft_agentic_stop\eval.jsonl` | 规范完整轨迹 + finalization-only 样本 |
| manifest | `data\novel_eval\sft_agentic_stop\manifest.json` | 记录来源、数量、样本类型和 held-out test 排除状态 |

构造规则：

- assistant 工具轮只保留一个 `<tool_call>...</tool_call>`，去掉 `<think>`，与 `qwen3_nothink` 评测口径一致。
- `full_trace` 样本保留完整工具调用和工具响应轨迹，最后以 `<answer>...</answer>` 结束。
- `finalization_only` 样本只监督检索证据后的最终 `<answer>`，用于提升停止条件和答案格式收敛。
- 脚本会拒绝把 `data\novel_eval\test.jsonl` 作为 train/eval 输入。

`eval_hf_model.py` 会额外输出 `tool_fragment_present` 和 `tool_fragment_rate`。它们统计任何裸 `<tool_call>` 或 `</tool_call>` 片段，能识别“脚本没有统计到成对 tool call，但模型实际输出了残缺工具标签”的情况。

`--forbid-tool-tags` 可作为 guarded direct-answer 诊断开关，用于临时屏蔽 `<tool_call>` 和 `</tool_call>` token；不要用它替代主评测指标，否则会掩盖模型本身的协议遵循问题。

直接答案评测只衡量最终答案输出能力，不代表工具调用能力。如果这里 F1 低但 `answer_tag_rate` 达标、`tool_fragment_rate` 很低，应定位为无检索直答知识不足；如果 Agent loop 能检索到正确证据并给出答案，应以 Agent loop 的 `final EM/F1` 和 `hop_recall` 判断闭环效果。

## Step 4.5: 生成 SFT Agent loop 评测

Agent loop 评测会让模型每轮只生成一个 `<tool_call>` 或最终 `<answer>`。脚本会解析合法 JSON tool call，调用本地 `HybridRetriever.dispatch()`，把检索结果包装为 `<tool_response>` 后追加回消息历史。

先用旧 Agentic SFT 模型各跑 5 条 raw / normalized smoke，判断是否存在“原始一轮多 action 污染历史”的问题：

```powershell
cd E:\AI\AgenticRAG-RL\demo

python .\scripts\eval_hf_agentic.py `
  --model .\models\Qwen3-4B-Instruct-2507-Unsloth-SFT-merged `
  --data .\data\novel_eval\test.jsonl `
  --corpus .\data\novel\corpus.jsonl `
  --output .\results\sft_compare\sft_agentic_smoke_raw.jsonl `
  --template qwen3_nothink `
  --max-samples 5 `
  --max-turns 5 `
  --per-turn-max-new-tokens 256 `
  --top-k 3 `
  --temperature 0 `
  --repetition-penalty 1.05 `
  --action-history-mode raw

python .\scripts\eval_hf_agentic.py `
  --model .\models\Qwen3-4B-Instruct-2507-Unsloth-SFT-merged `
  --data .\data\novel_eval\test.jsonl `
  --corpus .\data\novel\corpus.jsonl `
  --output .\results\sft_compare\sft_agentic_smoke_normalized.jsonl `
  --template qwen3_nothink `
  --max-samples 5 `
  --max-turns 5 `
  --per-turn-max-new-tokens 256 `
  --top-k 3 `
  --temperature 0 `
  --repetition-penalty 1.05 `
  --action-history-mode normalized
```

`raw` 模式会把模型原始输出写入下一轮历史，兼容旧评测；`normalized` 模式会把解析出的第一个合法 `<tool_call>` 或 `<answer>` 写入下一轮历史，但仍在 `raw_turns` 保存原始输出和诊断字段。

正式评测建议使用 Agentic-stop merged model，并同时保留 `--action-history-mode normalized` 诊断：

```powershell

  --corpus .\data\novel\corpus.jsonl `
  --output .\results\sft_compare\sft_agentic_eval.jsonl `
  --template qwen3_nothink `
  --max-samples 50 `
  --max-turns 5 `
  --per-turn-max-new-tokens 256 `
```

## Step 5: 计算对比指标

对 base 和 SFT 的预测结果计算同一组指标：

```powershell
python .\scripts\compare_predictions.py `
  --base .\results\sft_compare\base_predictions.jsonl `
  --sft .\results\sft_compare\sft_direct_predictions.jsonl `
  --output .\results\sft_compare\summary.json
```

`compare_predictions.py` 默认要求 base 和 SFT 的 `question` 集合完全一致。如果 base/SFT 来自不同语料、不同测试集或历史残留文件，脚本会直接失败并报告 `base_only_count` / `sft_only_count`，避免生成误导性 summary。只有在明确要做交集诊断时，才显式增加：

```powershell
--allow-question-mismatch
```

建议统计字段：

```text
count
base_avg_em
sft_avg_em
base_avg_f1
sft_avg_f1
base_answer_tag_rate
sft_answer_tag_rate
base_tool_call_rate
sft_tool_call_rate
base_tool_fragment_rate
sft_tool_fragment_rate
```

其中：

- `EM`：预测答案和标准答案完全一致。
- `Token F1`：预测答案和标准答案的 token overlap。
- `answer_tag_rate`：输出是否包含 `<answer>...</answer>`。
- `tool_call_rate`：单次生成输出是否包含 `<tool_call>{...}</tool_call>`。
- `tool_fragment_rate`：单次生成是否出现任何 `<tool_call>` 或 `</tool_call>` 片段，包括残缺标签。
- `valid_tool_call_rate`：Agent loop 中是否至少产生过一次合法工具调用。
- `hop_recall`：Agent loop 检索到的 chunk 是否覆盖 gold chunks。

direct-answer 校准模型的最低格式目标是 `answer_tag_rate >= 0.95`、`tool_fragment_rate <= 0.05`、`valid_tool_call_rate = 0`。Agentic SFT 模型的首要目标不是只提高 direct-answer F1，而是让模型学会稳定遵守 ReAct / tool-calling / `<answer>` 协议，因此 Agent loop 格式指标必须单独报告。

## Step 6: 可选 LLM-as-Judge

中文小说问答可能存在答案表述差异，EM / F1 会低估正确答案。可以在 base 和 SFT 预测上再跑真正的 LLM-as-Judge。

当前 Judge 使用所选 OpenAI-compatible 在线接口，默认 provider 为 `newapi`，默认模型为：

```text
gpt-5.5
```

运行前确认 `.env` 中已填写：

```text
NEWAPI_API_KEY=...
NEWAPI_BASE_URL=https://api.6i2.com/v1
NEWAPI_JUDGE_MODEL=gpt-5.5
```

命令模板：

```powershell
uv run python .\scripts\run_llm_judge.py `
  .\results\sft_compare\base_predictions_as_eval.json `
  --output .\results\sft_compare\base_judged.json `
  --judge-model gpt-5.5 `
  --max-concurrency 5

uv run python .\scripts\run_llm_judge.py `
  .\results\sft_compare\sft_direct_predictions_as_eval.json `
  --output .\results\sft_compare\sft_direct_judged.json `
  --judge-model gpt-5.5 `
  --max-concurrency 5
```

`--max-concurrency` 控制同时发起的 Judge 请求数，默认值是 `5`。如果遇到接口限流、超时或失败数上升，先降到 `1` 或 `2`。

Judge 会为每条样本输出：

```text
correctness
faithfulness
answer_format
overall
verdict
rationale
```

其中：

- `correctness`：预测答案是否与标准答案或别名语义等价。
- `faithfulness`：如果有证据文本，预测是否被证据支持；如果没有证据文本，则基于标准答案一致性做保守判断。
- `answer_format`：原始输出是否包含清晰最终答案，尤其是否遵守 `<answer>...</answer>`。
- `overall`：综合分。

注意：`run_llm_judge.py` 读取的是项目 eval JSON 结构，不是任意 JSONL 预测文件。`eval_hf_model.py` 已经会自动生成如下结构的 `_as_eval.json` 文件：

```json
{
  "summary": {},
  "results": [
    {
      "question": "...",
      "gold": "...",
      "prediction": "...",
      "evidence": [],
      "gold_chunks": [],
      "retrieved_chunk_ids": []
    }
  ]
}
```

## 推荐最终报告格式

训练前后对比结果建议整理成表格：

| 模型 / 评测线 | EM | F1 | `<answer>` 命中率 | tool fragment 率 | Agent `valid_tool_call_rate` | `hop_recall` | 备注 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| Qwen3-4B-Instruct-2507 / direct-answer | 待填 | 待填 | 待填 | 待填 | 不适用 | 不适用 | 训练前 |
| Qwen3-4B-Instruct-2507-Unsloth-SFT-direct-merged / direct-answer | 待填 | 待填 | 待填 | 待填 | 不适用 | 不适用 | direct-answer 校准后 |
| Qwen3-4B-Instruct-2507-Unsloth-SFT-merged / Agent loop | 待填 | 待填 | 待填 | 待填 | 待填 | 待填 | Agentic 工具闭环 |

结论应优先回答：

```text
1. SFT 后是否更稳定输出 <answer>？
2. direct-answer 输出是否仍出现裸 <tool_call> / </tool_call> 片段？
3. direct-answer EM / F1 是否相对 base 提升？
4. Agent loop 是否能稳定输出合法 <tool_call> 并召回 gold chunks？
5. 如果 F1 没提升，是检索没搜到、格式错误、无检索知识不足，还是答案表述差异？
```
