# 术语表

## corpus

原始语料集合。原项目里主要是上市公司半年报 PDF。

## chunk

切块后的最小检索单元。一个 chunk 既是检索索引的基本对象，也是 hop 标注指向的证据单位。

## index

为了检索 chunk 而建立的结构。这个工作台先保留三类概念：

- `BM25`：关键词匹配
- `dense`：向量语义匹配
- `hybrid`：多路融合

## hop

回答最终问题所必须经过的一步证据跳转。多跳不等于“问题看起来复杂”，而是“缺少任意一跳都无法完成答案推导”。

## plan

Planner 给出的子问题与工具调用序列。

## evidence

Executor 执行后收集到的证据。它不是最终答案，而是后续 Verifier 和 Synthesizer 的输入。

## trajectory

模型完成一轮搜索和回答的完整轨迹。对 SFT 来说它是监督样本，对 RL 来说它是 rollout。

## reward

用于 RL 的评分信号。原文重点讨论：

- correctness
- faithfulness
- hop matching
- LLM judge

## PEV

`Planner -> Executor -> Verifier` 的循环框架。原文再接一个 `Synthesizer` 输出最终答案。

## replan feedback

Verifier 告诉 Planner“为什么当前证据还不够”的反馈。原文实验说明，这部分质量比“单纯多迭代几轮”更重要。

## hop-aware 指标

用于定位多跳系统失败位置的指标，而不是只给一个最终分数。

## Oracle trajectory

直接从 gold hop 结构构造的理想搜索轨迹。优点是质量高，缺点是只覆盖“理论正确路径”。

## reward hacking

模型找到了“能拿高 reward 但违背真实目标”的捷径。本文最典型的例子就是：不搜索，直接编一个和问题词面接近的答案。
